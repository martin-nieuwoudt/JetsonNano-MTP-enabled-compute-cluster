Phase 1: Operating System Upgrades & Golden Master ProvisioningAcquire Base Image: Download the Qengineering Jetson Nano Ubuntu 20.04 Bare Image onto a local machine. This specific distribution patches L4T drivers onto an updated Ubuntu base while entirely stripping heavy pre-installed machine learning frameworks (Alternative: Pythops Jetson-Image Builder for a fully stripped headless build).Initialize Primary Node: Flash the downloaded bare image onto a single SD card (≥32 GB recommended) utilizing tools such as BalenaEtcher or Raspberry Pi Imager.Disable GUI Environment: Boot Nano #1, establish an SSH or terminal connection, and execute sudo systemctl set-default multi-user.target. Reboot the hardware to permanently disable desktop architecture loading, reducing the baseline system RAM footprint by approximately 400 MB.Execute System Updates: Execute sudo apt update && sudo apt upgrade -y to modernize existing package repositories.Configure Upgrade Parameters: Open the manager via sudo nano /etc/update-manager/release-upgrades. Alter the configuration to Prompt=normal. Save and exit the file.Initiate Headless Upgrade: Run the official server-distribution upgrade command sudo do-release-upgrade and follow terminal prompts. This natively upgrades the infrastructure to headless Ubuntu 22.04 LTS, instating Python 3.10 and GCC 11/12 compatibility without breaking underlying NVIDIA hooks.Establish Network Template: Configure the netplan/NetworkManager file to assign the SD card a clean template identity, verify the static IP is functioning, and power down the node.Create Golden Master Clone: Remove the provisioned SD card and mount it within the Windows workstation. Open a WSL or Git Bash terminal, identify the drive (e.g., /dev/sdX), and generate a snapshot clone using the command: sudo dd if=/dev/sdX of=/mnt/c/Users/YourUsername/Desktop/Jetson_22.04_Headless_Master.img bs=4M status=progress.Deploy Clones: Sequentially insert blank SD cards and write the master image to each utilizing the command: sudo dd if=/mnt/c/Users/YourUsername/Desktop/Jetson_22.04_Headless_Master.img of=/dev/sdX bs=4M status=progress.Phase 2: Swarm Network & RPC CompilationAssemble Hardware: Insert the cloned SD cards into the Nanos. Connect all active nodes directly to the Gigabit network switch, and boot the infrastructure.Define IP Variables: Verify each Nano possesses a static IP address ranging sequentially from 192.168.1.51 through 192.168.1.62 (accommodating up to two additional nodes). Node 10 (192.168.1.60) is initialized purely for continuous compute output, not network failover. Physically label each Nano with its assigned IP.Establish Secure Shell: Generate an SSH key on Windows via ssh-keygen -t ed25519. Copy the key to each Nano executing ssh-copy-id jetson@192.168.1.XX to enable passwordless transmission.Create Automation Script: Save setup_nano.sh on the Windows PC containing an array mapping all target IP addresses and the default username.Execute Remote Installs: Deploy the bash script to iterate through the target IP array via SSH. The script must execute sudo apt-get update && sudo apt-get install -y build-essential cmake git libcurl4-openssl-dev to install modern build tools.Map CUDA Environments: The deployment script must systematically append export PATH=/usr/local/cuda/bin:$PATH and export LD_LIBRARY_PATH=/usr/local/cuda/lib64:$LD_LIBRARY_PATH to ~/.bashrc on every node.Compile Collaborative Engines: The script must clone the llama.cpp repository, construct a build directory, and execute the compilation targeting the Maxwell architecture: cmake .. -DGGML_CUDA=ON -DGGML_RPC=ON -DCMAKE_CUDA_ARCHITECTURES=53 followed by cmake --build . --config Release.Automate Background RPC Services: The script must generate a systemd service file at /etc/systemd/system/llama-rpc.service to launch the RPC server automatically on boot:ExecStart=/home/jetson/llama.cpp/build/bin/rpc-server --host 0.0.0.0 --port 11434Restart=alwaysEnable Services: Execute sudo systemctl daemon-reload, enable, and start on the llama-rpc.service across all nodes.Verify Memory Baselines: SSH into an active node and execute free -h to verify 3.0–3.2 GB of free RAM (OS overhead natively constrained to 0.8–1.0 GB).Phase 3: Windows Master Orchestrator SetupCompile Windows Engine: Clone the llama.cpp repository natively on Windows. Construct the build directory and execute cmake .. -DGGML_CUDA=ON -DGGML_RPC=ON followed by cmake --build . --config Release. Verify functionality via .\bin\Release\llama-cli.exe --help.Provision Local Model Storage: Download target GGUF models directly to C:\Models\. Weights are exclusively stream-allocated to Nanos at boot; NFS/SMB protocols are prohibited. Primary target: Gemma 4 12B Q8_0 (~13–14 GB). Heavy alternative: Qwen 2.5 32B Instruct Q4_K_M (~20 GB).Install LiteLLM: Execute pip install litellm on Windows. Create litellm_config.yaml containing model list arrays mapping both the localized swarm and the RTX 3090 environments.Configure IDE Execution Parameters: Create .vscode/tasks.json in the primary workspace. Define discrete tasks to launch sub-clusters, start the LiteLLM router, and initialize the primary boot sequence ("🚀 Boot Local AI Cockpit").Phase 4: Activation & IDE IntegrationBoot the Swarm: Power on the network switch. Systemd automates RPC server initialization across all active Nanos.Launch Matrix Core: In VS Code, execute the compiled task sequence. The master initialization string must broadcast to all IPs utilizing the --rpc flag (e.g., .\bin\Release\llama-cli.exe --model C:\Models\gemma-4-12b-it-Q8_0.gguf --rpc 192.168.1.51:11434,192.168.1.52:11434... --ctx-size 8192 --server --port 8080).Verify Cluster Health: Query curl http://localhost:8080/v1/models to confirm node communication, followed by curl http://localhost:4000/v1/models to verify visibility within the LiteLLM router.Register IDE Endpoints: Navigate to VS Code Copilot Chat settings and establish a Custom OpenAI-Compatible Endpoint pointed to http://localhost:4000/v1.Assign Model Workloads: Configure the IDE to route inline autocomplete processes to the local RTX 3090 (diffusion-gemma) and sidebar chat operations directly to the Jetson swarm (gemma-4-swarm).Memory Budget DataVariantFile SizeCluster RAM UsedFree for KV CacheDeployment ViabilityGemma 4 12B Q4_K_M (4-bit QAT)~7 GB~7 GB~23-29 GBOptimalGemma 4 12B Q8_0 (8-bit)~13–14 GB~14 GB~16-22 GBSweet SpotGemma 4 12B F16 (unquantized)~24–26 GB~25 GB~5-11 GBMarginally ViableQwen 2.5 32B Q4_K_M~20 GB~20 GB~12-18 GBHighly ViableCommand R 35B Q4_K_M~22.5 GB~22.5 GB~7.5-13.5 GB
